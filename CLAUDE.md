# 기획 Tool 프로젝트 — Claude 컨텍스트

이 디렉토리는 ** 기획 tool ** 프로젝트의 기획 문서 관리 허브입니다.
휴맥스 IT(Humax IT)가 제공하는 디자인 시스템을 기반으로 PRD, 와이어프레임, 화면흐름도, 미팅노트, 결정로그를 작성·관리합니다.

├── _템플릿/                        ← 새 문서 작성 시 복사해서 사용

    └── [프로젝트명]/              ← 폴더명은 영문 권장 (예: CBAM_SaaS)
        ├── 00_inputs/             ← 프롬프트로 받은 입력 파일(참고 캡처 등) 보관 (화면/주제별 하위폴더)

## 문서 작성 규칙

### 파일 네이밍
- PRD: `PRD_v{버전}_{개요명}.md` + 동명 `.html` (예: `PRD_v1_전체개요.md` / `.html`)
- 화면흐름도: `화면흐름_{기능명}_v{버전}.md` + 동명 `.html` (예: `화면흐름_CBAM전체_v1.md` / `.html`)
- 기획화면(HTML, 개별): `SCR-{화면ID}_{화면명}.html` (예: `SCR-CBAM-001_벤치마크.html`)
- 기획화면(HTML, 통합본): `PROTO-{프로젝트코드}_v{버전}.html` (예: `PROTO-CBAM_v1.html`)
- 미팅노트: `YYYY-MM-DD_{회의주제}.md`
- 결정로그: `결정로그.md` (파일 하나에 누적, 최신순)
- 작업로그: `작업로그.md` (파일 하나에 누적, 최신순 — 요청→계획→진행→결과를 작업 단위로 추적)
- 히스토리 뷰어: `히스토리_뷰어.html`

> ⚠️ **와이어프레임(.md) 문서는 더 이상 작성하지 않는다.** 화면 단위 산출물은 아래 "기획화면 HTML" 로 일원화되었다.
> PRD·화면흐름도는 같은 이름의 `.md`/`.html` 두 파일을 함께 유지하며, 내용 수정 시 **양쪽을 동기화**한다.

### 버전 관리
- 파일명에 `_v{숫자}` 표기
- 소규모 수정 → 동일 파일 내 변경 이력 추가
- 대규모 변경 → 새 파일 생성 (기존 파일 유지)

### 문서 상태 뱃지
모든 문서 상단에 반드시 포함:
- `🟡 초안` — 작성 중
- `🔵 검토 중` — 피드백 수렴 중
- `🟢 확정` — 최종 승인 완료
- `🔴 폐기` — 더 이상 유효하지 않음

## ⛓️ 산출물 동기화 & 변경 전파 (필수 프로토콜)

> 이 섹션은 **어떤 스킬·명령으로 문서를 만들거나 수정하든** 반드시 적용된다.
> 사용자 입력으로 PRD·기획화면·화면흐름·미팅노트(결정로그) 중 **하나라도 바뀌면**,
> 영향받는 다른 문서를 함께 갱신하고 그 사실을 사용자에게 알려야 한다.

### 규칙 0 — 기본 작업 흐름은 **화면 우선(Screen-First)**

기획은 **화면을 먼저 설계·반복하고, 화면이 확정되면 그 화면을 기준으로 PRD·이력에 역전파**하는 순서를 기본으로 한다. (기획자가 가장 빠르고 정확하다고 판단한 흐름)

```
[1단계 · 화면 반복]  02_기획화면에서 SCR(.html 프로토타입 + 동반 .md)을 작성
   · 화면 요구사항·수정요청은 SCR 동반 .md의 「화면 요구사항 & 변경 요청」에 먼저 기록
   · 화면(.md/.html)만 빠르게 반복 수정 — 이 단계엔 PRD·화면흐름 전파 보류
   · 화면 상태: 🟡 초안 → 🔵 검토 중
        │
        ▼  화면 상태가 🟢 확정 되면 (또는 사용자가 "전파" 명시)
[2단계 · 확정→역전파]  확정된 화면을 기준으로 PRD·화면흐름·결정로그에 1회 일괄 전파
```

- **전파 시점**: 화면 상태가 `🟢 확정`이 되는 순간(또는 사용자 명시) 한 번에 전파한다. 반복 단계에서는 PRD를 건드리지 않아 화면을 빠르게 돌린다.
- **신규 프로젝트**: 최소 PRD 골격(범위·대상 사용자·화면 목록 수준)만 먼저 잡고 → 화면 설계로 들어간다. PRD 상세(기능 명세·데이터)는 화면 확정 후 채운다.
- **예외**: 화면에서 비롯되지 않은 직접적 PRD·정책 변경(미팅 결정 등)은 종전대로 규칙 2 매트릭스를 따라 즉시 전파한다.

### 규칙 1 — 모든 산출물은 `.md`(원본·이력) ↔ `.html`(뷰어·프로토타입) **쌍**으로 존재한다

| 단계 | `.md` (결정·이력의 단일 원천) | `.html` (뷰어 / 프로토타입) |
|------|------------------------------|---------------------------|
| `01_PRD` | `PRD_v{버전}_{개요명}.md` | 동명 `.html` (사이드바 목차 뷰어) |
| `02_기획화면` | `SCR-{화면ID}_{화면명}.md` ← **신설** | `SCR-{화면ID}_{화면명}.html` (기획문서 + 프로토타입) |
| `03_화면흐름도` | `화면흐름_{기능명}_v{버전}.md` | 동명 `.html` (Mermaid 뷰어) |
| `04_히스토리` | `작업로그.md` + `결정로그.md` + `미팅노트/*.md` | `히스토리_뷰어.html` |

- **`.html` 만 만들고 `.md`를 만들지 않는 것을 금지한다.** 결정 내용·변경 이력은 항상 `.md`가 단일 원천(Source of Truth)이며, `.html`은 이를 반영한다.
- `02_기획화면`의 HTML은 "기획문서 + 프로토타입" 두 뷰를 가진다. 화면의 **결정·변경 이력**은 동반 `.md`(`SCR-*.md`)에 기록하고, HTML 본문은 그 결과를 반영한다. (동반 `.md`는 `_템플릿/기획화면_개별_템플릿.md` 사용)
- 한쪽을 수정하면 **반드시 쌍의 다른 쪽도 동기화**한다.

### 규칙 2 — 변경 전파 의존성 매트릭스

한 문서가 바뀌면 아래 "→" 대상 문서의 정합성을 **반드시 점검·갱신**한다.

| 변경된 문서 | 함께 점검/갱신해야 하는 문서 |
|------------|---------------------------|
| **미팅노트** | → 결정로그(D-### 신규/갱신) → 그 결정이 닿는 PRD·기획화면·화면흐름 |
| **결정로그(D-###)** | → PRD(기능·범위·사용자스토리) → 기획화면(해당 SCR) → 화면흐름(해당 흐름) |
| **PRD** | → 기획화면(영향 SCR의 기능 명세·데이터) → 화면흐름(화면 목록·전이) → 결정로그(변경 사유) |
| **기획화면(SCR)** | → PRD(기능 목록·데이터 명세 정합) → 화면흐름(진입/전이/예외) → 동반 `.md`·결정로그 |
| **화면흐름** | → PRD(화면 목록) → 기획화면(진입경로·전이) → 결정로그 |

> 💡 **화면 우선 흐름의 주 경로는 「기획화면(SCR) → PRD·화면흐름·결정로그」 행**이다(규칙 0). 화면 확정 시 이 행을 따라 역전파한다. 나머지 행(미팅노트·결정로그·PRD 출발)은 화면에서 비롯되지 않은 변경에 적용한다.

### 규칙 3 — 변경 시 실행 절차 (화면 우선 기준, 반쯤 강제화)

작업 요청(프로젝트 등록·화면 추가·기능 수정 등)이 들어오면 아래 **A → B** 순서로 진행한다.
전체는 **작업로그의 [요청 → 계획 → 진행 → 결과]** 흐름과 대응한다.

**0. 작업 개시 (요청·계획)** — 해당 프로젝트 `04_히스토리/작업로그.md`에 `T-###` 블록을 **최신순(위로) 신설**한다.
상태 `🟡 계획`, 요청 내용 요약, **「작업 계획」 체크리스트**(아래 단계에서 손댈 문서 나열)를 채운다. (작업로그 없으면 `_템플릿/작업로그_템플릿.md`로 생성)

#### A단계 — 화면 반복 (전파 보류)
1. **요구사항 기록** — 화면 요구사항·수정요청을 해당 `02_기획화면/SCR-*.md`의 **「화면 요구사항 & 변경 요청」**에 먼저 적는다(상태 `🟡 대기`). 화면이 없으면 `_템플릿/기획화면_개별_템플릿.md`로 신설.
2. **화면 반복 (진행)** — SCR `.md/.html` **쌍**을 수정한다. 반영한 요청은 `🟢 반영`으로 바꾸고 SCR 「변경 이력」에 행을 추가한다. 화면 상태는 `🟡 초안`→`🔵 검토 중`. 작업로그 상태도 `🔵 진행 중`으로. **이 단계에선 PRD·화면흐름으로 전파하지 않는다.**

#### B단계 — 확정 → 역전파 (화면 상태 `🟢 확정` 시, 또는 사용자가 "전파" 명시)
3. **영향 범위 통지** — > 🔔 "이 화면 확정으로 다음 파일에 전파합니다: `PRD`, `화면흐름`, `결정로그` …"
4. **역전파 (연쇄 갱신)** — 확정 화면을 기준으로 의존성 매트릭스에 따라 PRD(기능 목록·데이터)·화면흐름(화면 목록·전이)·결정로그를 갱신한다. 각 문서의 `.md`↔`.html` 쌍도 동기화한다.
5. **이력 기록** — 변경된 각 문서 상단 **「변경 이력」 표에 행 추가**, 의사결정 동반 시 **`결정로그.md`에 `D-###` 최신순 추가**(관련 미팅노트·화면 링크 포함).
6. **상태·버전 갱신** — 상태 뱃지·`v{버전}` 조정.
7. **정합성 확인** — 내부 링크 경로·파일명·용어가 실제 파일과 일치하는지 점검.
8. **작업 마감 (결과)** — 작업로그 `T-###`를 `🟢 완료`로, **완료일·「결과/변경된 파일」 링크·관련 D-###**을 채우고 인덱스 표를 갱신한다.

> 사용자가 "전파하지 말라"고 하면 **A단계까지만**(화면 반복) 진행하고 B단계는 보류한다. 화면에서 비롯되지 않은 직접 PRD·정책 변경은 규칙 0 예외에 따라 A단계 없이 바로 B단계 매트릭스를 적용한다.

> 🤖 보조 장치: `02_기획화면` 등 산출물 파일을 Edit/Write 할 때 `.claude/settings.json`의 PostToolUse 훅이 위 체크리스트를 리마인드한다. 훅은 강제가 아닌 알림이며, 최종 책임은 이 프로토콜 준수에 있다.

## ID 체계

프로젝트 간 구분을 위해 화면·기능 ID에는 **프로젝트 코드**를 넣는다. (예: CBAM)

| 종류 | 형식 | 예시 |
|------|------|------|
| 화면 (개별) | `SCR-{프로젝트}-{3자리}` | `SCR-CBAM-001` |
| 통합 프로토타입 | `PROTO-{프로젝트}_v{버전}` | `PROTO-CBAM_v1` |
| 기능 | `F-{프로젝트}-{2자리}` | `F-CBAM-01` |
| 결정 | `D-{3자리}` | `D-001` |
| 작업 | `T-{3자리}` | `T-001` |
| 사용자스토리 | `US-{2자리}` | `US-01` |
| 비즈니스 규칙 | `BR-{2자리}` | `BR-01` |

> **문서 내 로컬 ID**: 개별 기획화면 HTML의 "화면 기능 명세"·"부록" 안에서는 해당 화면에 한정된
> `F-01`, `BR-01`처럼 프로젝트 코드 없는 짧은 번호를 쓴다. PRD·화면흐름도 등 **여러 화면을 아우르는
> 문서**에서는 반드시 프로젝트 코드를 포함한 전체 ID(`F-CBAM-01`, `SCR-CBAM-001`)를 사용한다.

## 디자인 원칙

1. **명확성** — 시멘틱 컬러 사용, 액션 우선순위 명시
2. **간결성** — 시각 요소 최소화, 1~2단 계층
3. **신뢰성** — 상태 색 명확 구분 (성공/경고/오류)
4. **접근성** — WCAG 2.1 AA (색대비 4.5:1, 키보드 네비, ARIA)
5. **반응형** — 데스크탑 우선, Bootstrap 5 그리드 기준

### 히스토리 뷰어 HTML 생성 규칙

- 새 프로젝트의 `04_히스토리/` 폴더에 **히스토리 뷰어**를 만들 때는 `_템플릿/히스토리_뷰어_템플릿.html`을 기준으로 작성한다.
- 템플릿 내 주석 `[A]` ~ `[E]` 위치에 실제 미팅노트·결정로그 내용을 채운다:
  - `[A]` — 미팅노트 목록 아이템 블록 (왼쪽 패널)
  - `[B]` — 미팅노트 원문 마크다운을 담는 `<textarea>` 블록
  - `[C]` — JS `MEETINGS` 객체에 회의 항목 추가
  - `[D]` — 결정로그 인덱스 테이블 행
  - `[E]` — 결정로그 아코디언 카드 블록
- `[프로젝트명]` 플레이스홀더를 실제 프로젝트명으로 교체한다.
- 미팅노트·결정 건수에 맞게 탭의 `badge-count` 숫자를 업데이트한다.
- CSS 디자인 토큰(`:root` 변수)과 JS 핵심 로직은 수정하지 않는다.

### 기획화면 HTML 파일 생성 규칙

- 기획화면을 만들 때는 **HTML과 동반 `.md`를 한 쌍으로** 만든다. (`SCR-*.html` + `SCR-*.md`)
  - 동반 `.md`는 `_템플릿/기획화면_개별_템플릿.md`를 복사해 작성하며, 화면의 **요구사항·결정·변경 이력의 단일 원천**이다. 화면 우선 흐름에서 요구사항·수정요청은 이 `.md`의 「화면 요구사항 & 변경 요청」에 **먼저** 기록한다(규칙 0).
  - HTML은 기획문서 + 프로토타입 두 뷰를 담고, 요구사항·결정·이력은 `.md`와 동기화한다. (위 "산출물 동기화 & 변경 전파" 프로토콜 적용)
- 새 기획화면 HTML은 **얇은 템플릿 `_템플릿/기획화면_개별_템플릿.html`을 복제**해 시작한다. 이 템플릿은 **공유 자산을 링크만** 한다:
  - 스타일 `_디자인시스템/cbam.css` + 프레임워크 JS `_디자인시스템/cbam-proto.js` (둘 다 `../../` 상대경로). **이 둘은 화면에 다시 써넣지 않는다(재생성 금지).**
  - 화면 본문은 `_템플릿/컴포넌트_카탈로그.md`의 클래스·마크업으로 **조립**한다. 거대한 `Layout_Rules.md`를 통째로 읽지 말 것. 인라인 `style`은 최소화한다.
- **화면 우선(Screen-First) 흐름** (규칙 0):
  - 복제 후 **[🖥️ 프로토타입] 뷰를 실제 화면으로 채우는 것이 1차 작업**이다. 화면 요구사항·수정요청은 동반 `SCR-*.md`의 「화면 요구사항 & 변경 요청」에 먼저 기록하고 반복한다.
  - **[📋 기획 문서] 뷰는 빈 캔버스 그대로 둔다.** 프로토타입을 채울 때 이 영역은 **읽지도·수정하지도 않는다.** 화면이 `🟢 확정`되거나 **사용자가 작성을 요청할 때만**, 빈 캔버스(템플릿의 `▼ 기획 문서 작성 영역 ▼` 주석 위치)에 doc-sidebar 목차 + 5개 섹션(1.서비스 개요 / 2.화면 목록(IA) / 3.화면 기능 명세 / 4.데이터 명세 / 5.부록)을 구성한다.
    - 기획 문서 작성 시 **서비스 개요 구성**: 상태 뱃지·버전 → 메타 블록(서비스·화면ID·화면타입·진입경로·접근권한) → 배경 및 목적 → 변경 이력. (PRD↔기획화면 역할 분리 준수 — KPI·사용자 스토리는 PRD에만)
- 템플릿 복제 시 교체할 항목:
  - `<title>`·app-logo·topbar의 `[PLACEHOLDER]`(화면ID·화면명·프로젝트·메뉴경로·사용자)
  - 하단 **inline `<script>`의 화면별 설정**: `NOTES_STORAGE_KEY='cbam-notes-{ID}'`, `NOTES_EMBED_ID='embeddedNotes{ID}'`, `SAVE_FILENAME`, 그리고 하단 `<script id="embeddedNotes{ID}">` — **화면마다 고유**(안 그러면 localStorage 노트가 섞임). `SCREEN_IDS`(서브탭 순서), `INIT_NOTES`(노트, body는 `<ul><li>` 형식).
  - `[SCREEN-CONTENT]`에 프로토타입을 카탈로그로 조립.
- **수정/재생성하지 않는 것**: `cbam.css`, `cbam-proto.js`(탭·정렬·노트·모달·노트 앵커링). **디자인 변경은 `cbam.css` 한 곳만** 고치면 전 화면에 반영된다.
- **단독 파일로 외부 공유**가 필요할 때만 `_템플릿/기획화면_통합_템플릿.html`(임베드본)을 쓴다.

### 다중 화면 통합 standalone 생성 후처리 (필수)

여러 SCR 파일을 하나의 standalone HTML로 병합할 때마다 아래 **3가지 버그**가 반드시 발생한다. 병합 직후 순서대로 수정한다.

#### 1. `<style>` 이중 중첩 제거
병합 스크립트가 각 화면의 인라인 `<style>`을 `<style id="style-scr001">` 래퍼로 감싸면서 `<style>` 태그가 중첩된다. CSS 변수(`--dark`, `--bg` 등)가 파싱되지 않아 LNB 배경 등이 사라진다.

```
# 잘못된 구조 (각 화면마다 반복)
<style id="style-scr001">
<style>          ← 내부 중첩 태그
...CSS...
</style>
</style>         ← 고아 닫기 태그

# 올바른 구조
<style>
...CSS...
</style>
```

수정: 각 화면의 외부 `<style id="style-scr001">` 줄과 마지막 고아 `</style>` 줄을 삭제한다.

#### 2. cbam-proto.js 내 DOM 직접 접근 → null 에러 방지
`cbam-proto.js`가 `<head>`에 인라인되면 `document.querySelector('.proto-screens-wrap')` 등이 DOM 로드 전 실행되어 null 에러로 스크립트가 중단된다.

수정: cbam-proto.js 인라인 블록에서 아래 두 곳을 `DOMContentLoaded`로 감싼다.

```javascript
// ① contextmenu 이벤트 바인딩
// 변경 전: document.querySelector('.proto-screens-wrap').addEventListener('contextmenu', e => { ... });
// 변경 후:
document.addEventListener('DOMContentLoaded', function() {
  document.addEventListener('contextmenu', function(e) {
    if (!e.target.closest('.proto-screens-wrap')) return;
    // ... 기존 핸들러 내용
  });
  document.getElementById('widgetCtxAdd').addEventListener(...)
  document.getElementById('widgetCtxEdit').addEventListener(...)
  document.getElementById('widgetCtxDel').addEventListener(...)
}); // end DOMContentLoaded

// ② 초기화 코드
// 변경 전: if(!loadNotes()){ ... } renderNotesList('list');
// 변경 후:
document.addEventListener('DOMContentLoaded', function() {
  if (typeof INIT_NOTES !== 'undefined' && !loadNotes()) { ... }
  if (typeof renderNotesList === 'function') renderNotesList('list');
});
```

#### 3. `toggleNotes` 전역 함수 충돌 → 활성 패널 기준으로 override
각 화면이 각자의 패널 ID(`descPanel`, `notePanel`, `notesPanel`)를 타겟하는 `toggleNotes`를 정의하고, 마지막 정의가 전역을 덮어써 다른 화면의 Description 패널을 닫지 못한다.

수정: 화면 전환 로직(`switchScreen`) 스크립트의 IIFE 내부 **마지막 줄**에 다음을 추가한다.

```javascript
// ── 통합뷰어 전용: toggleNotes를 활성 패널 기준으로 최종 override ──
window.toggleNotes = function() {
  var activeScreen = document.querySelector('.screen-panel.active') || document;
  var panel = activeScreen.querySelector('.lnb-desc-panel');
  var btn = activeScreen.querySelector('.btn-notes-toggle');
  if (!panel) return;
  var open = panel.classList.toggle('open');
  if (btn) btn.classList.toggle('active', open);
  document.querySelectorAll('.note-badge-overlay').forEach(function(el) { el.remove(); });
};
```

> 이 세 수정은 **병합 방식이나 화면 수와 무관하게** 통합 standalone을 만들 때마다 반드시 적용한다.

### 캡처(참고 화면) 저장 & 에이전트 분업 (효율 규칙)

- **프롬프트로 입력 파일(참고 캡처 이미지·자료 등)이 주어지고 저장이 필요하다고 판단되면 `projects/{프로젝트}/00_inputs/`에 저장한다.** (화면 캡처는 `00_inputs/SCR-{ID}/`처럼 화면/주제별 하위폴더로.) 캡처는 화면의 **시각적 단일 원천**이며, 디스크에 있으면 서브에이전트도 `Read`로 볼 수 있다. (채팅에 첨부만 된 이미지는 서브에이전트가 못 봄.)
- **무거운 파일 작업(대량 편집·산출물 생성)은 서브에이전트에 위임**하고, 메인은 **이미지 해석·스펙/영향범위 확정·병합·브라우저 검증**만 한다 → 메인 컨텍스트 경량화로 토큰↓. 비용의 대부분은 "메인이 큰 파일을 매 턴 캐시 재읽기"에서 나온다.
  - 빌더 서브에이전트에는 **캡처 경로 + 완결 스펙 + `컴포넌트_카탈로그.md` 경로**를 준다.
- **단일 파일 동시 편집 금지**(충돌). 화면을 병렬 생성할 땐 화면당 별 파일로 만들고 메인이 병합한다.
- 가능하면 `sed`로 라인 범위를 잘라 파일을 생성/추출한다(본문을 컨텍스트에 안 읽어 토큰 절감 — 공유 CSS/JS 추출이 그 예).

### PRD / 화면흐름도 HTML 뷰어 생성 규칙

- PRD·화면흐름도는 `.md`(원본)와 `.html`(사이드바 목차형 뷰어)을 **한 쌍**으로 만든다.
- PRD HTML 뷰어는 **`_템플릿/PRD_템플릿.html`을 복제**해 시작하고, `[[...]]` 플레이스홀더와 제목·목차(`doc-nav-item`)·본문을 실제 내용으로 교체한다. (기존 PRD가 있는 프로젝트라면 그 폴더의 최신 `PRD_*.html`을 기준 파일로 복제해도 된다. 화면흐름도는 `_템플릿/화면흐름도_뷰어_템플릿.html` 사용)
- `.md` 내용과 `.html` 내용은 항상 **동기화**한다. 한쪽만 수정하지 않는다.
- 복제 시 CSS 디자인 토큰(`:root`)과 핵심 JS(네비게이션·Mermaid 렌더 로직)는 수정하지 않는다.

### 화면흐름도 Mermaid 작성 규칙 — 데이터 체크 (필수)

Mermaid 10.x에서 `Syntax error in text` 오류를 막기 위해 다음을 **반드시** 지킨다.

- **줄바꿈은 `<br/>` 만 사용한다.** 노드/엣지 라벨 안의 `\n`(리터럴 백슬래시-n)은 줄바꿈이 아니라 문법 오류를 일으킨다.
  - ❌ `M1[SCR-CBAM-001\nCBAM 벤치마크 조회]`
  - ✅ `M1["SCR-CBAM-001<br/>CBAM 벤치마크 조회"]`
- **괄호 `()` · 콜론 `:` · 따옴표 · `#` 등 특수문자를 포함한 라벨은 큰따옴표로 감싼다.**
  - ✅ `R11["수출량 상세 조회 모달<br/>(F-CBAM-08)"]`
- 화면흐름도 HTML은 `mermaid.initialize({ startOnLoad: false, securityLevel: 'loose' })` + **수동 렌더 가드**를 사용한다. 가드는 다음 두 가지를 보장한다(템플릿에 내장):
  1. 작성 실수로 `\n`을 쓴 경우 렌더 직전 자동으로 `<br/>` 로 치환한다(`sanitizeMermaid`).
  2. 다이어그램별 `mermaid.parse()`로 사전 검증해, 오류가 난 블록만 빨간 오류 박스로 표시하고 **나머지 다이어그램·페이지는 정상 렌더**되도록 격리한다.
- 기준 파일/템플릿을 복제할 때 위 초기화 옵션과 `sanitizeMermaid`·`renderMermaidAll` 함수는 수정하지 않는다.

### 용어·내부 링크 일관성 점검

- 파일을 생성·수정한 뒤에는 PRD·화면흐름도·기획화면 사이의 **내부 링크 경로와 파일명이 실제 파일과 일치하는지 확인**한다. (예: 링크는 `SCR-CBAM-002_출하데이터수집관리.html` 인데 파일명이 다르면 깨진 링크가 된다.)
- 한 산출물 세트 안에서 동일 대상의 **용어를 통일**한다. (예: "출하" vs "출고" 혼용 금지)

### Type A 리스트 화면 — 컬럼 정렬 규칙

- **Type A(목록 조회) 화면의 모든 `<th>` 컬럼에는 반드시 오름차순/내림차순 정렬 기능을 포함한다.**
- 구현 방식:
  - 각 `<th>`에 `class="sortable"` 추가 및 `onclick="sortTable(colIdx)"` 핸들러 연결
  - 헤더 텍스트 뒤에 `<span class="sort-icon none"></span>` 삽입
  - 미정렬: `⇅` (회색), 오름차순: `▲` (파란색), 내림차순: `▼` (파란색)
  - 같은 헤더 재클릭 시 방향 전환 (asc → desc → asc)
- JS 데이터 배열 기반 화면(동적 렌더링): `sortBy(key)` + `getFiltered()` 내 sort 로직 + `renderSortIcons()` 패턴 사용
- 정적 HTML 행 기반 화면: DOM 행 재정렬 방식 (`sortState`, `sortTable(i)`, `updateSortIcons()` 패턴) 사용
- 숫자·금액 컬럼은 수치 비교(`parseFloat`), 텍스트 컬럼은 한국어 로케일(`localeCompare(…, 'ko')`) 비교
- "-"(미입력) 값은 숫자 정렬 시 항상 맨 아래로 배치

## PRD ↔ 기획화면 역할 분리

> 작업 **순서**는 화면 우선(규칙 0)이지만, 두 문서가 담는 **내용의 역할 분리**는 그대로다.
> 화면 단계에서 쌓는 「화면 요구사항 & 변경 요청」은 **이 화면에 대한 구체적 UI·기능 요청**이고,
> 화면 확정 시 이를 **PRD의 기능 목록·데이터 명세로 추상화·통합**한다. (사용자 스토리·KPI 등 제품 차원 항목은 여전히 PRD에만 둔다.)

PRD와 HTML 기획 문서는 담는 내용을 아래와 같이 분리한다.

| 항목 | PRD (.md) | 기획화면 HTML |
|------|-----------|--------------|
| 배경 및 목적 | ✅ 전체 기능 관점 | ✅ 이 화면 관점 (간략히) |
| 목표 & KPI | ✅ | ❌ |
| 대상 사용자 | ✅ | ❌ |
| 사용자 스토리 | ✅ | ❌ |
| 기능 목록 & 우선순위 | ✅ 여러 화면 포함 | ❌ |
| 비기능 요구사항 | ✅ | ❌ |
| 화면 메타 (ID·타입·진입경로·권한) | ❌ | ✅ |
| UI 구성요소 & 기능 상세 | ❌ | ✅ |
| 데이터 명세 | ❌ | ✅ |
| 비즈니스 규칙 | ❌ | ✅ 화면별 |
| Open Issues | ✅ 전체 | ✅ 화면별 |

- HTML 기획 문서의 **서비스 개요**에는 핵심 사용자·KPI·사용자 스토리를 작성하지 않는다. 해당 내용은 PRD에만 작성한다.
- HTML 기획 문서의 **부록**에는 이 화면에 종속된 비즈니스 규칙과 Open Issues만 담는다.

## Claude 행동 지침

- 문서 생성 시 항상 해당 템플릿 파일을 먼저 읽고 구조를 따른다.
- **작업은 화면 우선(Screen-First)으로 진행한다(규칙 0).** 화면을 먼저 만들고 요구사항·수정요청을 `SCR-*.md`의 「화면 요구사항 & 변경 요청」에 먼저 쌓아 화면을 반복하다가, 화면이 `🟢 확정`되면 그때 PRD·화면흐름·결정로그로 역전파한다. 반복 단계에서 임의로 PRD를 먼저 고치지 않는다.
- 새 프로젝트 생성 시 `README.md`의 프로젝트 목록 표와 이 파일의 "진행 중인 프로젝트" 표 양쪽에 행을 추가한다. 프로젝트 폴더명은 영문을 권장한다(예: `CBAM_SaaS`). **시작은 최소 PRD 골격(범위·대상 사용자·화면 목록)만** 잡고 바로 화면 설계로 들어가며, PRD 상세는 화면 확정 후 채운다. `04_히스토리/`에는 결정로그·히스토리 뷰어와 함께 **`작업로그.md`도 `_템플릿/작업로그_템플릿.md`로 초기화**한다.
- **모든 작업 요청은 작업로그에 `T-###` 블록으로 추적한다.** 위 "산출물 동기화 & 변경 전파 § 규칙 3"의 0단계 → A단계(화면 반복) → B단계(확정 역전파)를 따른다: 착수 시 `🟡 계획`으로 블록을 열고, 진행하며 체크리스트·상태를 갱신, 확정·전파까지 끝나면 `🟢 완료`로 마감하고 인덱스·결과물 링크를 채운다.
- HTML 프로토타입은 Bootstrap 5 + Reactstrap 스타일로 작성하고, CDN 링크를 사용해 단독 실행 가능하게 한다.
- 화면흐름도 다이어그램은 Mermaid `flowchart TD` 문법을 사용하고, 위 "화면흐름도 Mermaid 작성 규칙"(줄바꿈 `<br/>`, 특수문자 라벨 따옴표, 렌더 가드)을 반드시 지킨다.
- PRD·화면흐름도의 `.md`와 `.html`은 항상 함께 갱신한다.
- 결정로그는 최신 항목이 위로 오도록 누적 기록한다.
- 디자인 시스템 참조가 필요한 경우 `_디자인시스템/` 하위 파일을 직접 읽는다.
